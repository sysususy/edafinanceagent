# edafinanceagent 零基础复现教程

这是一份面向零基础学习者的完整教程，目标是让你在没有 NVIDIA GPU 的普通电脑上，也能把这个项目的核心部分真正复现出来。

本教程默认你的环境和我们实际验证过的环境一致：

- Windows 主机
- `WSL 2 Ubuntu`
- 没有 `NVIDIA GPU`
- 有 `OpenRouter API Key`

## 1. 先明确项目目标

这个项目不是单一脚本，而是三段式工程：

1. 多 Agent 主系统  
路径：`Financial-MCP-Agent/`

2. A 股 MCP 数据服务  
路径：`a-share-mcp-is-just-i-need/`

3. 金融新闻训练链路  
路径：根目录下的 `data_process.py`、`train_qwen_sentiment.py`、`train_qwen_risk.py`

如果你是零基础，推荐的顺序一定是：

1. 先跑通多 Agent 主系统
2. 再理解数据清洗与训练链路
3. 最后再考虑云端 LoRA / GRPO / vLLM

## 1.1 先用一句话理解整套系统

如果只用一句话概括，这个项目做的事情是：

> 把“用户输入一个股票问题”拆成多个专业分析子任务，让不同 Agent 各自调用金融数据工具完成分析，最后再汇总成一份结构化报告。

对应到工程实现上，就是：

- `Financial-MCP-Agent/` 负责多 Agent 编排
- `a-share-mcp-is-just-i-need/` 负责把金融数据包装成工具
- 训练链路负责给新闻分析部分补充更专业的情感与风险判断能力

这三个部分分别对应了 Agent 系统里最常见的三类能力：

- 编排能力：谁先做、谁后做、谁汇总
- 工具能力：怎么查真实数据，而不是只靠模型瞎猜
- 模型能力：怎么让某个具体任务的判断更稳定、更可控

## 1.2 理论和工程是怎么一一对应的

很多人学 Agent 项目时，容易把“论文概念”和“代码文件”分开看。这个项目其实很适合建立对应关系。

### 对应关系 1：ReAct

理论上，ReAct 的核心思想是：

- 先思考
- 再决定是否调用工具
- 拿到工具结果后继续推理

工程上，在这个项目里对应的是：

- Agent 接收自然语言任务
- Agent 选择合适的 MCP 工具
- 用工具返回的真实数据继续生成分析结论

### 对应关系 2：MCP

理论上，MCP 的核心价值是：

- 把外部能力标准化成可调用工具
- 让模型不用知道底层实现细节，只要知道工具名和输入输出

工程上，在这个项目里对应的是：

- `a-share-mcp-is-just-i-need/` 暴露出统一的金融工具
- `Financial-MCP-Agent/src/tools/mcp_client.py` 负责把这些工具接入主系统
- Agent 不直接操作 Baostock，而是通过 MCP 工具间接获取数据

### 对应关系 3：LangGraph

理论上，LangGraph 适合解决：

- 多步骤任务
- 多角色协作
- 状态需要在节点之间传递

工程上，在这个项目里对应的是：

- 基本面、技术面、估值、新闻四个 Agent 分别执行
- 汇总 Agent 负责拿到前面所有结果再生成最终报告
- 执行日志和状态对象帮助追踪整条链路

### 对应关系 4：SFT / LoRA

理论上，SFT 想解决的是：

- 让模型学会按指定格式完成任务

LoRA 想解决的是：

- 在不全量训练大模型的情况下，用更低成本做领域适配

工程上，在这个项目里对应的是：

- 把“新闻情感分类”和“风险评分”改写成生成式对话任务
- 用 `Assistant: 1~5` 的形式让模型学会输出评分
- 后续如果有云端 GPU，再把这套样本和脚本迁移到正式训练环境

## 2. 为什么本教程适合无 GPU 环境

没有 NVIDIA GPU，并不意味着这个项目做不了。

你仍然可以完整完成：

- MCP 工具化
- LangGraph 多 Agent 编排
- OpenRouter 模型接入
- 新闻数据清洗
- SFT 样本构造
- Tokenizer / Label Masking 验证

你当前不适合本机做的是：

- `Qwen3-8B` 正式 LoRA 训练
- `vLLM` 本地推理部署
- `GRPO` 正式强化学习训练

因此，本教程采用两段式目标：

### 第一阶段：主系统跑通

目标是：

- 能输入股票名称或代码
- 能调用 MCP 工具查数据
- 能生成 Markdown 报告

### 第二阶段：训练链路理解

目标是：

- 看懂数据结构
- 看懂 prompt 如何构造
- 看懂为什么只训练答案 token
- 发现训练脚本里的真实工程问题

## 2.1 为什么零基础最适合先跑主系统，再看训练

这是一个非常重要的学习顺序。

如果一开始就盯着训练脚本，很容易卡在：

- 模型路径
- 显存不足
- tokenizer 细节
- 数据样本格式

但这些问题对“先理解整个系统在做什么”并不是最关键的。

更合理的顺序是：

1. 先让系统真正跑起来
2. 理解多 Agent 为什么要拆成四类分析
3. 再去理解新闻模型为什么需要单独训练

也就是说：

- 主系统解决“整体架构是否成立”
- 训练链路解决“局部能力是否可增强”

这也是很多真实工程团队的工作方式：先把主链路打通，再优化局部模型能力。

## 9. 修复 Baostock 并发问题

原项目在四个 Agent 并发调用时，容易出现：

- `用户未登录`

根本原因是：

- Baostock 登录态是进程级全局状态
- 多个并发登录/登出会互相打断

### 9.1 修改文件

文件：

`a-share-mcp-is-just-i-need/src/utils.py`

#### 增加导入

```python
import threading
```

#### 增加全局锁

```python
_BAOSTOCK_SESSION_LOCK = threading.RLock()
```

#### 替换 `baostock_login_context`

```python
@contextmanager
def baostock_login_context():
    """上下文管理器，处理Baostock登录和登出，抑制标准输出消息"""
    with _BAOSTOCK_SESSION_LOCK:
        original_stdout_fd = sys.stdout.fileno()
        saved_stdout_fd = os.dup(original_stdout_fd)
        devnull_fd = os.open(os.devnull, os.O_WRONLY)

        os.dup2(devnull_fd, original_stdout_fd)
        os.close(devnull_fd)

        logger.debug("Attempting Baostock login...")
        lg = bs.login()
        logger.debug(f"Login result: code={lg.error_code}, msg={lg.error_msg}")

        os.dup2(saved_stdout_fd, original_stdout_fd)
        os.close(saved_stdout_fd)

        if lg.error_code != '0':
            logger.error(f"Baostock login failed: {lg.error_msg}")
            raise LoginError(f"Baostock login failed: {lg.error_msg}")

        logger.info("Baostock login successful.")
        try:
            yield
        finally:
            original_stdout_fd = sys.stdout.fileno()
            saved_stdout_fd = os.dup(original_stdout_fd)
            devnull_fd = os.open(os.devnull, os.O_WRONLY)

            os.dup2(devnull_fd, original_stdout_fd)
            os.close(devnull_fd)

            logger.debug("Attempting Baostock logout...")
            bs.logout()
            logger.debug("Logout completed.")

            os.dup2(saved_stdout_fd, original_stdout_fd)
            os.close(saved_stdout_fd)
            logger.info("Baostock logout successful.")
```

### 9.2 为什么要这么改

因为一次完整的数据调用应该被看成：

`登录 -> 查询 -> 登出`

这个过程必须串行化，否则多个 Agent 会互相把 Baostock 会话踢掉。

### 这一步体现了什么工程能力

这是一个很典型的“理论上能跑，工程上不稳定”的例子。

理论上，多个 Agent 并发调用工具没有问题。
但工程上，Baostock 的登录态不是线程隔离的，而是共享全局状态。

所以这里的修复本质上不是“为了让代码不报错”，而是在做：

- 并发安全分析
- 共享状态保护
- 会话边界控制

这类问题在真实项目里非常常见，也非常适合作为你简历里的“稳定性优化”案例。

## 15. 训练链路里真正重要的知识点

### 15.1 这不是普通分类头训练

情感与风险任务都被包装成了生成式对话任务：

- `System`
- `User`
- `Assistant`

模型不是输出分类 logits，而是学习生成最后的评分数字。

### 15.2 为什么要做 Label Masking

训练时不是整段文本都算 loss，而是：

- system prompt 不算
- few-shot 示例不算
- user 输入不算
- 只让最后的 `Assistant: 2` 这种答案 token 参与 loss

### 15.3 真实踩到的两个坑

#### 坑 1：token 对齐问题

例如：

- `"2"` -> `[17]`
- `" 2"` -> `[220, 17]`
- `"Assistant: 2"` -> `[71703, 25, 220, 17]`

如果只按字符位置切分，可能保留到的是空格 token，而不是真正的标签 token。

#### 坑 2：长 Prompt 截断

原模板很长：

- system prompt 很长
- few-shot 示例很多
- 新闻摘要也很长

当 `max_length=512` 时，最后的答案 `Assistant: 2` 可能直接被截掉，导致训练信号消失。

### 15.4 `left padding` 的理解

如果设置：

```python
tokenizer.padding_side = "left"
```

那么：

- 左边是 padding
- 真实文本在右边

所以判断真实 token 区间时，不能按右侧 padding 的思路处理。

## 15.5 从“理论名词”到“你真正做了什么”

如果你学到这里，建议把这些概念翻译成自己的工程语言：

- ReAct  
  不是抽象名词，而是“Agent 会根据问题决定是否调工具”

- MCP  
  不是协议名词，而是“把查数据能力包装成模型能调用的工具”

- LangGraph  
  不是框架名词，而是“把多 Agent 任务拆成节点和状态流”

- SFT  
  不是简单地“喂数据训练”，而是“让模型学会按指定对话格式输出评分”

- Label Masking  
  不是技巧堆砌，而是“只让真正的答案 token 参与学习”

当你能把理论名词翻译成这些工程动作时，说明你已经不是“会抄教程”，而是真的开始理解系统了。
